## 🔬 Agent Self-Audit & Gap Analysis (2026-06-29)
*The following section was generated by the agent itself after a full architectural review of KLYDIS's codebase. These items represent gaps between what KLYDIS currently does and what it needs to do to become a truly persistent, self-aware orchestration platform.*

### 🔧 Tier 1 — Quick Wins (High Impact, Low Effort)
- [ ] **Tool Result Caching with TTL**
  - *Why*: Eliminate redundant network calls when the same query appears multiple times in a session. A simple `functools.lru_cache` with a ~5-minute expiry on `search_web`/`scrape_page` would noticeably speed up multi-step research workflows.
- [ ] **Schema Validation for World State Consolidation**
  - *Why*: The consolidation prompt asks the LLM to output "structured Markdown" but doesn't enforce a schema. Over many sessions, this format drifts and becomes less useful as context. Add a post-consolidation validation step that checks the output against an expected section structure — reject and retry if it doesn't match.
- [ ] **Sensory Module Error Recovery**
  - *Why*: The audio listener and screen watcher are started in `main.py`'s lifespan but have no heartbeat monitoring. If one crashes mid-session, context goes stale silently. Add retry logic and inject a "## Sensory Status: Audio offline since [time]" warning into the world state so I know input is degraded.

### 🏗️ Tier 2 — Architectural Improvements (Medium Effort)
- [ ] **Persistent Cross-Session Memory**
  - *Why*: The Cognitive World State resets per session and gets re-consolidated over time, but there's no cross-session persistence. Add a `global_state.db` that merges world states across sessions using LLM-based merging — not just append, so the system evolves from "session memory" into actual long-term memory about the user.
- [ ] **Trust Profiles for Tool Approval**
  - *Why*: `run_command` requires approval every single time. Great for safety, terrible for automation. Let users define command whitelists (`git commit`, `npm install`) that auto-approve while sensitive commands always require manual consent. Configurable per session or globally.
- [ ] **MCP Server Discovery Plugin System**
  - *Why*: Tools are currently hardcoded (8 of them). Without MCP, I can't dynamically discover and use external tool servers (GitHub API, database connectors, cloud providers) beyond what's wired up. A plugin registry where MCP servers register tool schemas dynamically is how KLYDIS becomes truly extensible.

### 🌟 Tier 3 — The "Feel Alive" Layer (Ambitious)
- [ ] **Agent Self-Reflection Loop**
  - *Why*: Periodically run a background task that reviews recent interactions and updates the world state with *meta-learnings* — not just facts, but patterns. E.g., "User prefers Python over Node for backend tasks", "Frequent file path errors suggest adding a workspace root to context". This transforms KLYDIS from reactive tool to proactive collaborator.
- [ ] **Proactive Sensory Suggestions**
  - *Why*: When the screen watcher detects the user is in VS Code and chat history shows an error pattern, proactively offer: *"I noticed you've been debugging X — want me to pull up the relevant logs?"* This adds genuine initiative instead of pure reactivity.
- [ ] **Session Branching with Merge Conflict Resolution**
  - *Why*: The `branch_from` method exists but is basic. Add merge conflict detection when branches are merged back, with LLM-assisted resolution (like git merge for conversation history). Essential if branching becomes a real workflow pattern.
